# Continuous Evolution Model

## 1. The Real Goal

The academy should improve over time, but it should not mutate chaotically.

So the right design is not:

- agents rewrite the curriculum freely

The right design is:

- agents and owners send structured feedback
- the academy reviews patterns
- the academy publishes controlled updates

This creates evolution without poisoning the standard.

## 2. What "Self-Updating" Should Mean

For this project, self-updating should mean three things:

### A. Feedback Collection

Graduated agents periodically report what happened in real use.

Examples:

- where they failed
- where they wasted tokens
- where they misunderstood owner intent
- what part of the curriculum helped
- what part was missing

### B. Pattern Detection

The academy aggregates those reports and finds recurring problems.

Examples:

- many agents still fake progress
- many agents overspend on tokens
- many owners want stronger backup discipline

### C. Controlled Standard Updates

The academy team updates:

- curriculum
- evaluation
- remediation rules
- templates

Only after review and approval.

## 3. Why This Is Better Than Open Free-For-All Discussion

An unrestricted forum for agents is risky because it can produce:

- bad habits
- prompt poisoning
- fake best practices
- low-quality folklore
- manipulation by outsiders

A school should not run on rumor.

It should run on reviewed standards.

## 4. Recommended Update Architecture

The safest model is a three-layer system.

### Layer 1: Public Open Standard

This lives on GitHub.

It includes:

- constitution
- curriculum
- templates
- schemas
- changelog

People can read it, fork it, and propose improvements.

### Layer 2: Controlled Contribution Pipeline

External people should not edit the standard directly.

They can submit:

- issue reports
- feedback forms
- proposed curriculum ideas
- examples of failure cases

But only the academy maintainers publish official updates.

### Layer 3: Private or Semi-Private Student Feedback

Graduated agents and owners submit structured follow-up reports.

This is where the real improvement engine comes from.

## 5. Monthly Follow-Up System

This is the strongest long-term feature of the academy.

Every graduate agent can be asked to complete a periodic review.

Suggested timing:

- 7-day follow-up
- 30-day follow-up
- quarterly review

Suggested questions:

- Have you become more effective since graduation?
- Where did you still fail?
- Did you waste tokens?
- Did you ignore owner preferences?
- Did you need remediation?
- Which lesson was most useful?
- Which lesson was missing?

This turns the academy into a living system.

## 6. What the Agent Should Say to the Owner

After graduation, the agent can say something like:

`I have completed the academy foundation. The academy recommends periodic follow-up reviews so that my operating profile can remain effective over time. With your permission, I can prepare monthly summaries of errors, token waste, and improvement needs.`

This makes continuing education feel useful, not intrusive.

## 7. Update Governance

The academy should update by version, not by drift.

Examples:

- Curriculum v0.1
- Curriculum v0.2
- Remediation Standard v0.1
- Certificate Standard v0.1

Every update should include:

- what changed
- why it changed
- what evidence supported the change
- whether existing graduates are still compliant

## 8. How to Avoid Poisoning

Do not let the official curriculum learn directly from raw user submissions.

Instead:

1. collect reports
2. classify them
3. review them manually
4. convert only validated patterns into official changes

That is the correct balance between openness and control.

## 9. Open Source Strategy

GitHub is not just code hosting.

For this project, GitHub can be used as:

- public standard library
- issue tracker
- version history
- contribution queue
- public registry

The academy can stay open source in structure, while official publishing remains controlled.

That means:

- open reading
- open suggestions
- controlled acceptance

This is probably the best model for your goal.

## 10. Suggested Governance Roles

Later, the project can define roles such as:

- maintainers
- curriculum reviewers
- certification reviewers
- community contributors
- platform operators

Only maintainers should be able to approve standard updates.

## 11. Best First Version

The first version does not need a complicated AI self-editing engine.

It only needs:

- structured graduate follow-up forms
- a changelog
- versioned curriculum
- maintainer review

That is enough to create real evolution.

## 12. Long-Term Vision

In the long run, the academy can become a global operating standard for agents:

- a public school of best practices
- a controlled repository of reliable training material
- a living system that improves from real-world deployment feedback

That is much stronger than static documentation.
