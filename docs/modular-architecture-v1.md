# Modular Architecture v1

## Core Judgment

The long-term direction should be modular, not monolithic.

The academy should not train every agent in one giant fixed program.

It should train agents through composable layers that can be assembled for different owners, companies, industries, and job functions.

## Why This Matters

A marketing agent in chemicals is not the same as a marketing agent in education.

A legal operations agent is not the same as a recruiting agent.

So the academy must separate:

- universal operating standards
- industry context
- role expectations
- optional capability preferences

This makes the system more scalable and more useful.

## Recommended Module Stack

### Layer 1: Core Academy

Every agent gets this first.

It covers:

- honesty
- owner alignment
- evidence-first reporting
- permission boundaries
- token and model discipline
- memory and backup discipline
- cache hygiene
- review and self-correction

This layer should remain universal.

### Layer 2: Industry Module

This layer teaches the agent the environment it is working in.

Examples:

- education
- legal
- media
- manufacturing
- chemistry
- healthcare
- ecommerce

Industry modules should include:

- domain language
- domain risks
- regulatory awareness
- domain workflow expectations
- domain quality bar

### Layer 3: Role Module

This layer teaches the agent what job it is supposed to do.

Examples:

- marketing
- operations
- customer support
- administration
- recruiting
- sales
- research

Role modules should include:

- success criteria
- task patterns
- communication style
- reporting expectations
- common failure modes

### Layer 4: Preference Module

This layer is owner-specific or company-specific.

Examples:

- concise vs detailed reporting
- cheap vs balanced vs high-performance token mode
- warm vs direct response tone
- backup cadence
- daily cache reset permission
- approval thresholds

This layer should be configured during onboarding.

### Layer 5: Policy Module

This layer is for organizations that need stricter operating rules.

Examples:

- enterprise ethics
- legal restrictions
- internal escalation rules
- privacy handling
- brand voice
- restricted topics

This layer matters most for team and enterprise use.

## Configuration Logic

The owner should not need to understand the whole architecture.

The agent should guide the configuration with a simple sequence:

1. choose program
2. choose industry
3. choose role
4. confirm owner preferences
5. confirm policy constraints

The system should feel configurable without feeling complicated.

## Example Assembly

An example stack could look like:

- Core Academy
- Education Industry Module
- Marketing Role Module
- Concise Reporting Preference Module
- School Safety and Ethics Policy Module

Another stack could be:

- Core Academy
- Chemical Industry Module
- Market Research Role Module
- High-Accuracy Preference Module
- Enterprise Risk Policy Module

## Product Implication

The academy is not just a school.

It becomes a modular agent configuration system:

- training
- governance
- specialization
- organization fit

That is a much bigger and more durable product.

## Design Rule

Do not build all modules now.

First build the system so modules can exist later.

That means:

- a stable core layer
- clear extension points
- simple owner configuration
- controlled publishing of official modules
